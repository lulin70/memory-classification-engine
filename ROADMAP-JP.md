## ビジョン

**MCE は AI Agent の標準記憶分類ミドルウェアです。**

**ChromaDB** がベクトルストレージの代名詞であるように、MCE は**記憶分類**の代名詞となることを目指します —— 任意の記憶システムに入るものを決める「セキュリティチェック」。

**製品ポジショニング**: *"あなたの Agent は Supermemory/Mem0/Obsidian で記憶を**保存**します。MCE は**何を**保存すべきか教えます。"*

**核心叙事**: 先に分類、後に保存。あなたが選ぶ。

## 戦略決定：純上流ルート (2026-04-19)

**決定文書**: [MCP_POSITIONING_CONSENSUS_v3.md](./docs/consensus/MCP_POSITIONING_CONSENSUS_v3.md)

### 何が変わったか

| 之前 (v0.2.0) | 之后 (v0.3.0+) |
|---------------|----------------|
| MCP: 11ツール（分類 + 全CRUD） | MCP: 4ツール（分類のみ） |
| "記憶分類エンジン" | "記憶分類**ミドルウェア**" |
| Supermemory / Mem0 と競合 | **補完**関係（下流顧客） |
| 内蔵 SQLite 必須 | StorageAdapter ABC 経由で委譲 |
| 叙事: "全部は覚えない" | 叙事: "**先分類、後保存**" |

### この決定の理由

1. **Supermemory は YC + Benchmark #1 + Cloudflare インフラ** — 保存で競争不可
2. **Mem0 は 18k Stars + ベクトル+グラフハイブリッド** — 保存は実績十分
3. **しかし、どちらも保存前に分類をしていない** — これがギャップ
4. **60%+ のメッセージは LLM 処理不要** — 保存に依存しない独立した価値

## 決定記録

### DR-001: なぜフルスタックではなく純上流を選んだか？

**選択肢**:
- A: フルスタック（Supermemory/Mem0 と競争）❌
- B: 純上流（分類のみ、保存は下流へ）✅
- C: デュアルモード（2パッケージ）❌

**決定**: ルート B — 純上流

**主要理由**:
1. 保存で競争不可能（Supermemory YC投資、Mem0 18k stars）
2. しかし誰も分類をうまくやっていない — 7件の競品記事が**全てこの次元を無視**
3. 「倉庫 vs セキュリティチェック」比喩は説得力があり差別化されている
4. コード59%減、テスト53%減、保守負担大幅軽減
