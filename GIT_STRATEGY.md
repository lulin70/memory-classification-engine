# Git 策略对比表

## 📋 10 个详细对比表

### 1. 核心原则对比 - 判断维度的总纲

| 维度 | 应该放入 Git | 不应该放入 Git | 决策依据 |
|------|-------------|---------------|----------|
| 必要性 | 项目核心代码和配置 | 临时文件和自动生成文件 | 项目运行和构建所必需 |
| 可重复性 | 源代码和配置模板 | 依赖包和构建产物 | 确保代码可以在任何环境中重建 |
| 版本控制 | 人工编写的文件 | 自动生成的文件 | 需要跟踪变更历史 |
| 安全性 | 公开的配置和代码 | 包含敏感信息的文件 | 避免泄露敏感信息 |
| 存储效率 | 小文件和文本文件 | 大文件和二进制文件 | 保持仓库大小合理 |

### 2. 代码文件对比 - 源代码、字节码、依赖包

| 文件类型 | 应该放入 Git | 不应该放入 Git | 决策依据 |
|---------|-------------|---------------|----------|
| 源代码 | ✅ .py, .js, .java 等 | ❌ 无 | 项目的核心逻辑，需要版本控制 |
| 字节码 | ❌ .pyc, .class 等 | ✅ 无 | 自动生成，可从源代码重建 |
| 依赖包 | ❌ node_modules, site-packages 等 | ✅ requirements.txt, package.json 等 | 依赖包体积大，可通过配置文件重建 |
| 配置文件 | ✅ config.yaml, .env.example 等 | ❌ .env, local_settings.py 等 | 配置模板需要版本控制，实际配置因环境而异 |

### 3. 配置文件对比 - 模板 vs 实际配置

| 配置类型 | 应该放入 Git | 不应该放入 Git | 决策依据 |
|---------|-------------|---------------|----------|
| 配置模板 | ✅ config.yaml, .env.example 等 | ❌ 无 | 提供配置示例，确保一致性 |
| 实际配置 | ❌ .env, local_settings.py 等 | ✅ 无 | 包含环境特定信息，可能包含敏感数据 |
| 构建配置 | ✅ webpack.config.js, Makefile 等 | ❌ 无 | 构建过程需要版本控制 |
| 部署配置 | ✅ docker-compose.yml, k8s 配置等 | ❌ 无 | 部署过程需要版本控制 |

### 4. 文档文件对比 - 产品文档 vs 过程文档

| 文档类型 | 应该放入 Git | 不应该放入 Git | 决策依据 |
|---------|-------------|---------------|----------|
| 产品文档 | ✅ README.md, API文档等 | ❌ 无 | 产品使用和开发指南需要版本控制 |
| 过程文档 | ✅ 设计文档, 架构文档等 | ❌ 会议记录, 个人笔记等 | 项目设计和决策需要版本控制，个人过程文档不需要 |
| 测试文档 | ✅ 测试计划, 测试用例等 | ❌ 测试报告, 覆盖率报告等 | 测试方法需要版本控制，测试结果是动态生成的 |
| 用户文档 | ✅ 用户手册, 教程等 | ❌ 无 | 用户指南需要版本控制 |

### 5. 数据文件对比 - 数据库结构 vs 数据文件

| 数据类型 | 应该放入 Git | 不应该放入 Git | 决策依据 |
|---------|-------------|---------------|----------|
| 数据库结构 | ✅ 迁移文件, 建表脚本等 | ❌ 无 | 数据库结构需要版本控制 |
| 数据文件 | ❌ 生产数据, 测试数据等 | ✅ 示例数据, 种子数据等 | 实际数据体积大，且可能包含敏感信息 |
| 存储文件 | ❌ 上传的文件, 媒体文件等 | ✅ 示例图片, 图标等 | 存储文件体积大，且是用户生成的 |
| 缓存文件 | ❌ .cache, __pycache__ 等 | ✅ 无 | 缓存文件是自动生成的，可重建 |

### 6. 测试文件对比 - 测试代码 vs 测试报告

| 测试类型 | 应该放入 Git | 不应该放入 Git | 决策依据 |
|---------|-------------|---------------|----------|
| 测试代码 | ✅ test_*.py, spec.js 等 | ❌ 无 | 测试逻辑需要版本控制 |
| 测试报告 | ❌ coverage.xml, test-results 等 | ✅ 无 | 测试报告是动态生成的 |
| 测试配置 | ✅ pytest.ini, jest.config.js 等 | ❌ 无 | 测试配置需要版本控制 |
| 测试数据 | ❌ 大量测试数据 | ✅ 小型测试数据 | 小型测试数据有助于测试重现，大型测试数据体积大 |

### 7. 构建部署对比 - CI/CD配置 vs 构建产物

| 构建类型 | 应该放入 Git | 不应该放入 Git | 决策依据 |
|---------|-------------|---------------|----------|
| CI/CD配置 | ✅ .github/workflows, .gitlab-ci.yml 等 | ❌ 无 | CI/CD配置需要版本控制 |
| 构建产物 | ❌ dist/, build/ 等 | ✅ 无 | 构建产物是自动生成的，可重建 |
| 部署脚本 | ✅ deploy.sh, k8s 配置等 | ❌ 无 | 部署过程需要版本控制 |
| 环境配置 | ✅ staging.yml, production.yml 等 | ❌ 无 | 环境配置需要版本控制 |

### 8. IDE 工具对比 - 项目配置 vs 个人偏好

| IDE 配置 | 应该放入 Git | 不应该放入 Git | 决策依据 |
|---------|-------------|---------------|----------|
| 项目配置 | ✅ .editorconfig, tsconfig.json 等 | ❌ 无 | 项目级配置需要团队共享 |
| 个人偏好 | ❌ .vscode/, .idea/ 等 | ✅ 无 | 个人IDE设置因用户而异 |
| 代码风格 | ✅ .eslintrc, .prettierrc 等 | ❌ 无 | 代码风格规则需要团队共享 |
| 调试配置 | ❌ launch.json, tasks.json 等 | ✅ 无 | 调试配置是个人偏好 |

### 9. 日志缓存对比 - 运行时文件

| 运行时文件 | 应该放入 Git | 不应该放入 Git | 决策依据 |
|---------|-------------|---------------|----------|
| 日志文件 | ❌ logs/, *.log 等 | ✅ 无 | 日志文件是运行时生成的，体积大 |
| 缓存文件 | ❌ .cache, __pycache__ 等 | ✅ 无 | 缓存文件是自动生成的，可重建 |
| 临时文件 | ❌ tmp/, *.tmp 等 | ✅ 无 | 临时文件是运行时生成的，不需要版本控制 |
| 会话文件 | ❌ *.sess, session/ 等 | ✅ 无 | 会话文件是运行时生成的，不需要版本控制 |

### 10. 敏感信息对比 - 安全风险项

| 敏感信息 | 应该放入 Git | 不应该放入 Git | 决策依据 |
|---------|-------------|---------------|----------|
| API密钥 | ❌ 实际API密钥 | ✅ 密钥模板, 环境变量示例 | 实际API密钥会导致安全风险 |
| 数据库凭证 | ❌ 实际数据库凭证 | ✅ 凭证模板, 环境变量示例 | 实际数据库凭证会导致安全风险 |
| 密码 | ❌ 任何密码 | ✅ 无 | 密码会导致安全风险 |
| 令牌 | ❌ 实际令牌 | ✅ 无 | 令牌会导致安全风险 |
| 证书 | ❌ 实际证书 | ✅ 证书请求文件 | 实际证书会导致安全风险 |

## 📊 产品文件 Git 管理策略

基于以上对比表，以下是作为产品的 Git 管理策略：

### 应该放入 Git 的文件

1. **核心代码文件**：
   - 所有的 Python 源代码文件
   - 配置文件模板（如 config.yaml）
   - 构建和部署脚本

2. **文档文件**：
   - README.md 和 README-EN.md
   - 设计文档和架构文档
   - API 文档
   - 测试用例和测试计划

3. **配置文件**：
   - CI/CD 配置文件（如 .github/workflows）
   - 项目级配置文件（如 .editorconfig）
   - 依赖管理文件（如 requirements.txt）

4. **测试文件**：
   - 测试代码（如 test_*.py）
   - 测试配置文件
   - 小型测试数据

5. **示例文件**：
   - 示例代码（如 examples/ 目录下的文件）
   - 示例配置

### 不应该放入 Git 的文件

1. **数据文件**：
   - 实际的记忆数据（如 data/ 目录下的文件）
   - 数据库文件（如 *.db）
   - 上传的文件和媒体文件

2. **运行时文件**：
   - 日志文件（如 logs/ 目录下的文件）
   - 缓存文件（如 __pycache__/）
   - 临时文件

3. **构建产物**：
   - 编译后的文件（如 .pyc）
   - 打包后的文件（如 dist/ 目录下的文件）

4. **敏感信息**：
   - 环境变量文件（如 .env）
   - 包含 API 密钥的文件
   - 包含数据库凭证的文件

5. **个人配置**：
   - IDE 配置文件（如 .vscode/, .idea/）
   - 个人调试配置

## 🎯 实施建议

1. **使用 .gitignore 文件**：
   - 配置一个全面的 .gitignore 文件，排除所有不应该放入 Git 的文件
   - 确保团队成员都遵循相同的 Git 策略

2. **分支管理**：
   - 使用主分支（main）存储稳定的代码
   - 使用开发分支（develop）进行开发
   - 使用特性分支（feature/*）开发新功能
   - 使用发布分支（release/*）准备发布

3. **提交规范**：
   - 使用清晰的提交消息格式
   - 每个提交只包含一个功能或修复
   - 提交前运行测试，确保代码质量

4. **代码审查**：
   - 使用 Pull Request 进行代码审查
   - 确保代码符合项目规范
   - 确保没有将不应该放入 Git 的文件提交

5. **安全管理**：
   - 使用环境变量存储敏感信息
   - 使用密钥管理服务管理 API 密钥
   - 定期检查仓库中是否包含敏感信息

通过遵循这些策略，可以确保 Git 仓库保持干净、高效，同时确保项目的安全性和可维护性。
