# 记忆分类引擎 - 产品需求文档

## 1. 用户故事

### 1.1 核心用户故事

#### 故事 1: 智能助手开发者
**作为** 智能助手开发者
**我希望** 集成一个记忆分类引擎到我的助手系统中
**以便** 助手能够记住用户的偏好和重要信息，提供更个性化的服务

#### 故事 2: 企业客服系统管理员
**作为** 企业客服系统管理员
**我希望** 系统能够智能记忆客户的历史需求和偏好
**以便** 客服代表能够快速了解客户背景，提供更高效的服务

#### 故事 3: 个人用户
**作为** 个人用户
**我希望** 与AI助手的对话能够保持连续性，不需要重复提供相同的信息
**以便** 获得更自然、更个性化的交互体验

### 1.2 详细用户场景

#### 场景 1: 代码风格偏好记忆
**用户**：开发者
**场景**：用户在与AI助手的对话中提到"我喜欢使用双引号而不是单引号"，助手将此信息分类为用户偏好并存储
**结果**：后续生成的代码中自动使用双引号，无需用户重复提醒

#### 场景 2: 业务事实记忆
**用户**：企业员工
**场景**：用户提到"我们公司有5个部门，其中技术部有20人"
**结果**：助手将此信息分类为事实声明并存储到语义记忆中，后续相关问题能够基于此信息回答

#### 场景 3: 决策记录记忆
**用户**：项目负责人
**场景**：团队讨论后达成一致"我们决定采用React作为前端框架"
**结果**：助手将此决策记录存储到情节记忆中，后续相关讨论能够参考此决策

#### 场景 4: 记忆管理
**用户**：系统管理员
**场景**：用户发现某个记忆信息过时或错误，希望删除或更新
**结果**：助手提供记忆管理界面，允许用户查看、编辑和删除存储的记忆

## 3. 技术架构

### 3.1 架构概述

记忆分类引擎采用模块化设计，由核心引擎、多层判断管道、存储层和工具层组成。系统以轻量级为设计原则，优先使用本机现有技术，确保在资源受限环境下也能正常运行。

### 3.2 技术选型

| 组件 | 技术方案 | 选型理由 |
|------|----------|----------|
| 核心语言 | Python 3 | 本机已安装，生态丰富，适合AI相关开发 |
| 规则引擎 | YAML配置 + 正则表达式 | 轻量级，易于配置和扩展 |
| 对话状态跟踪 | 内存状态机 | 简单高效，适合初期开发 |
| 程序性记忆存储 | YAML/JSON文件 | 轻量级，易于读写和版本控制 |
| 情节记忆存储 | SQLite + 向量嵌入 | 本机支持，无需额外安装，适合中小型应用 |
| 语义记忆存储 | SQLite + 关系模型 | 本机支持，无需额外安装，适合结构化数据 |
| 语义分类 | 本地小模型（如有）或API调用 | 灵活适配不同环境 |
| 接口形式 | Python SDK + REST API | 易于集成到不同系统 |

### 3.4 核心模块设计

#### 3.4.1 核心引擎
- **功能**：协调各模块工作，处理外部请求
- **接口**：
  - `process_message(message)`：处理用户消息，返回记忆分类结果
  - `retrieve_memories(query)`：根据查询检索相关记忆
  - `manage_memory(action, memory_id, data)`：管理记忆（查看、编辑、删除）

#### 3.4.2 多层判断管道
- **规则匹配层**：基于正则表达式和关键词匹配，识别明确的用户信号
- **结构分析层**：分析对话的交互模式，识别重复提问、方案接受/拒绝等模式
- **语义推断层**：调用模型进行语义分析，处理复杂的记忆识别

#### 3.4.3 记忆管理
- **去重与冲突检测**：避免重复记忆，处理记忆冲突
- **遗忘机制**：基于时间、频率和重要性的加权衰减算法
- **记忆压缩**：将细节信息压缩为模式，减少存储占用

#### 3.4.4 存储层
- **工作记忆**：内存存储，会话结束后清除
- **程序性记忆**：YAML/JSON文件存储，适合用户偏好、行为规则
- **情节记忆**：SQLite存储，适合决策记录、任务模式
- **语义记忆**：SQLite存储，适合事实声明、关系信息

### 3.5 数据模型

#### 3.5.1 记忆元数据
```json
{
  "id": "mem_20260403_001",
  "type": "user_preference",
  "tier": 2,
  "content": "使用双引号而不是单引号",
  "created_at": "2026-04-03T10:00:00Z",
  "updated_at": "2026-04-03T10:00:00Z",
  "last_accessed": "2026-04-03T10:30:00Z",
  "access_count": 5,
  "confidence": 1.0,
  "source": "rule:0",
  "context": "用户在讨论代码风格时提到",
  "status": "active"
}
```

#### 3.5.2 存储结构
- **程序性记忆**：YAML文件，按类型组织
- **情节记忆**：SQLite表，包含记忆元数据和向量嵌入
- **语义记忆**：SQLite表，包含实体-关系-属性结构

### 3.6 部署与集成

#### 3.6.1 部署方式
- **本地部署**：作为独立模块集成到现有系统
- **容器部署**：打包为Docker容器，方便在不同环境运行

#### 3.6.2 集成方式

**Layer 1: Python SDK**
- **Python SDK**：直接导入到Python项目中
- **REST API**：通过HTTP接口与其他语言系统集成

**Layer 2: MCP Server** ⭐ 近期重点
- **MCP Server**：符合 Model Context Protocol 标准的服务器
- **Claude Code 集成**：通过 MCP 配置直接集成
- **Cursor 集成**：通过 MCP 配置直接集成
- **OpenClaw 集成**：通过 CLI 适配器集成

**Layer 3: Framework Adapters**
- **LangChain**：提供 LangChain Tool 封装
- **CrewAI**：提供 CrewAI Tool 封装
- **AutoGen**：提供 AutoGen Agent 封装
- **其他框架**：可扩展的适配器架构

### 3.7 性能与安全

#### 3.7.1 性能优化
- **缓存机制**：频繁访问的记忆放入内存缓存
- **异步处理**：语义推断等耗时操作采用异步处理
- **批量操作**：支持批量记忆处理，减少I/O操作

#### 3.7.2 安全措施
- **数据加密**：敏感记忆信息加密存储
- **访问控制**：基于角色的访问控制
- **审计日志**：记录所有记忆操作，便于追溯

## 5. 验收标准

### 5.1 功能验收
- 能够正确识别和分类至少7种记忆类型
- 能够将记忆存储到对应的层级
- 能够根据查询检索相关记忆
- 能够自动衰减和淘汰低价值记忆
- 能够处理记忆冲突

### 5.2 性能验收
- 记忆分类响应时间不超过100ms
- 记忆检索响应时间不超过200ms
- 系统能够处理每分钟至少100次请求

### 5.3 可靠性验收
- 系统故障后记忆数据不丢失
- 能够恢复到故障前的状态
- 内存占用不超过500MB

## 6. 结论

记忆分类引擎是一个轻量级、模块化的系统，能够智能识别、分类和管理AI Agent的记忆信息。通过分层存储和遗忘机制，系统能够在保证重要信息不丢失的同时，有效控制存储容量和检索效率。

本设计充分考虑了本机现有技术环境，采用轻量级的存储方案，确保系统能够在资源受限的环境下正常运行。同时，系统具有良好的可扩展性，未来可以根据需求添加新的功能和支持更多的存储后端。

通过实施本计划，我们将构建一个功能完善、性能可靠的记忆分类引擎，为AI Agent提供更智能、更个性化的服务能力。
