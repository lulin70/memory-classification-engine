> **产品初心**: AI remembers you. Not the other way around.
> **核心使命**: 让 AI 记忆属于用户，可携带、可控制、可信任
> **当前版本**: v0.4.3 | **目标版本**: v1.0.0
> **制定日期**: 2026-04-26
> **更新日期**: 2026-04-26 (v2 — 调整阶段顺序)

## 一、从初心出发 — 我们要解决什么问题？

用户的核心痛点：
1. **AI 总是忘记我** — 每次新对话从零开始
2. **记忆被工具锁定** — 换工具丢记忆
3. **什么都存** — 噪声爆炸，检索无效
4. **记忆不属于我** — 数据在别人手里

CarryMem 的差异化价值：
- ✅ 自动分类（60%+ 零成本）— 已实现
- ✅ 语义召回（跨语言）— 已实现
- ✅ 可携带（export/import）— 已实现
- ⏳ **真正好用** — 还有距离

**"真正好用"的定义**：一个新用户在 5 分钟内安装、运行、看到效果，并在日常工作中持续依赖它。

## 三、为什么 PyPI 发布要后置？

### 反面教训

记忆系统是**信任敏感型产品**。用户把个人偏好、纠正、决策交给 CarryMem，这和普通工具库不同：

- 普通库：不好用就换，无损失
- 记忆系统：不好用 = **数据风险**，用户会永久离开

如果用户 `pip install carrymem` 后发现：
- 无加密 → "我的偏好数据是明文的？"
- 无备份 → "数据库损坏了，记忆全丢了？"
- 无远程模式 → "只能本地用？换台电脑就没记忆了？"
- 测试覆盖率低 → "这东西靠谱吗？"

**结论：PyPI 发布只有一次首秀机会，必须在产品值得信赖时才亮相。**

### 正面策略

早期通过 `git clone + pip install -e .` 分发给早期用户（开发者群体），他们容忍度高、反馈质量高。等产品真正成熟再上 PyPI，面向大众用户。

## 四、优化提升计划（调整后）

### Phase 1: 让人想用（v0.5.0）— 主动记忆推送

**目标**: CarryMem 不只是被动存储，而是主动为 AI 提供上下文

| 编号 | 任务 | 价值 | 工作量 |
|------|------|------|--------|
| 1.1 | **智能上下文注入** | AI 自动获得用户记忆，无需手动 recall | 中 |
| 1.2 | **记忆衰减与强化** | 常用记忆权重上升，过时记忆自然消退 | 中 |
| 1.3 | **查询缓存层** | 重复查询毫秒级响应 | 小 |
| 1.4 | **记忆合并与冲突解决** | 多 Agent 场景下记忆一致性 | 大 |
| 1.5 | **记忆版本化** | 修改可追溯，支持回滚 | 中 |

**核心设计 — 智能上下文注入**:
```python
# 当前：用户需要手动 recall
memories = cm.recall_memories(query="database")

# 目标：CarryMem 自动注入到 AI 对话
cm = CarryMem()
prompt = cm.build_system_prompt(context=current_conversation)
# → 自动包含最相关的 10 条记忆 + 5 条知识库条目
# → 按相关性、时效性、访问频率排序
```

**验收标准**: AI 对话中自动获得用户偏好，无需手动 recall

## 七、不做什么（明确边界）

| 不做 | 原因 |
|------|------|
| Web UI | 当前阶段 CLI + API 足够，Web UI 是 v2.0 的事 |
| 移动应用 | 非核心场景，记忆在服务端管理 |
| 云同步服务 | 与"数据属于用户"理念冲突，S3 适配器已满足需求 |
| LLM 训练 | CarryMem 是记忆层，不是模型训练工具 |
| 多租户 | 当前面向个人用户，企业功能是 v2.0 的事 |
