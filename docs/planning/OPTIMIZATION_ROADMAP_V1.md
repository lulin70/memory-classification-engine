## 一、初心锚定

| 维度 | 内容 |
|------|------|
| **产品定位** | 专业的 AI Agent 记忆分类引擎（非 AutoML、非 Context Optimizer） |
| **核心使命** | "Don't remember everything. Remember what matters." |
| **价值主张** | 实时记忆分类，60%+ 零 LLM 成本 |
| **技术底线** | 三层管道（规则→模式→语义），四级存储，绝不引入 Meta-Agent 复杂度 |

### 初心来源
基于 README.md 核心定义和与北大 MetaContextEngine 论文对比分析后的团队共识：
- 我们是 **Memory Classification Engine** — 记忆分类引擎
- 北大论文是 **Meta Context Engineering** — 元上下文工程（双层进化+AutoML）
- 两者名称相似但本质不同，必须坚守自己的定位

## 三、Phase 1 详细实施计划

### 3.1 查询效率优化 +30%

#### 3.1.1 技术方案（架构师设计）

```
优化路径:
├── A. SmartCache 命中率提升 (预期贡献: +15%)
│   ├── A1. 查询预热机制 — 启动时加载热点数据
│   ├── A2. 缓存策略调优 — LRU淘汰参数动态调整
│   └── A3. 多级缓存联动 — tier1→tier2→tier3穿透优化
│
├── B. 存储层索引优化 (预期贡献: +10%)
│   ├── B1. 记忆ID哈希索引 — O(1)查找替代线性扫描
│   ├── B2. 类型+时间复合索引 — 加速范围查询
│   └── B3. 关联关系预构建 — 减少运行时关联计算
│
└── C. 批量查询优化 (预期贡献: +5%)
    ├── C1. retrieve_memories 批量接口
    ├── C2. 并行层级查询 — tier2/tier3/tier4并发获取
    └── C3. 结果合并去重优化
```

#### 3.1.2 任务拆解（独立开发者）

| 任务ID | 任务描述 | 复杂度 | 预估工时 | 依赖 |

| T1.2 | SmartCache预热机制实现 | 中 | 1d | T1.1 |

| T1.5 | 记忆ID哈希索引实现 | 中 | 1d | 无(可并行) |
| T1.6 | 类型+时间复合索引 | 中 | 1d | T1.5 |

### 5.2 不通过条件

以下任一条件不满足则 **禁止发布**：
- 查询效率提升 <20%（目标30%，最低门槛20%）
- 存在 Critical 级别回归缺陷
- MCP Server 存在未处理的崩溃场景
- API文档覆盖 <90%
