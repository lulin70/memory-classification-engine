## 三、七成员投票

### 投票规则

- 每位成员对 15 项议题打分：5(最高优先) / 4(高优先) / 3(中等) / 2(低优先) / 1(暂缓)

## 五、共识讨论

### 议题 1：PyPI 发布时机

**分歧**: PM(2) 和 UX(2) 认为应延后，开发者(4) 和社区(4) 认为应同步推进。

**讨论**:
- PM: "产品成熟度 6/10，现在发布 = 让用户失望"
- 开发者: "PyPI 发布的工程量很小，和 CLI 增强不冲突"
- 社区: "早期用户通过 git clone 安装，PyPI 是降低门槛"
- 架构师: "可以分步：先做 CLI 增强，同时准备 PyPI，最后一起发布"

**共识**: PyPI 发布不作为独立阶段，而是作为 v0.8.0 的收尾动作。先完成 P0 用户体验改进，再发布到 PyPI。

### 议题 2：TUI vs Web UI

**分歧**: TUI 均分 3.86(P1)，Web UI 均分 2.43(P2)。

**讨论**:
- UX: "TUI 是可视化 3/10 → 7/10 的最快路径，textual 成熟且轻量"
- 架构师: "Web UI 引入前端技术栈，架构复杂度跳升"
- 开发者: "textual 开发效率高，1-2周可出 MVP"
- QA: "TUI 测试成本可控，Web UI 测试复杂度高"

**共识**: TUI 优先(P1)，Web UI 暂缓(P2)。TUI 验证需求后再考虑 Web UI。

### 议题 3：技术债务处理方式

**分歧**: QA(5) 和安全(5) 认为必须优先，UX(2) 认为不紧急。

**讨论**:
- QA: "D-3 MCP端到端测试、D-4 CLI测试缺失，新功能无法保证质量"
- 安全: "D-5 SQL注入误杀、D-6 路径遍历检测是安全漏洞"
- 开发者: "技术债务和新功能开发可以交替进行"

**共识**: 技术债务不单独设阶段，而是作为每个 P0/P1 任务的验收条件。具体：
- A1 增强 CLI → 必须补齐 CLI 测试 (D-4)
- A3 简化 MCP → 必须补齐 MCP 端到端测试 (D-3)
- A9 carrymem doctor → 必须修复安全检测问题 (D-5, D-6)

### 议题 4：Cursor 插件优先度

**分歧**: PM(4) 和 UX(4) 和社区(4) 倾向高优先，QA(2) 和安全(3) 倾向低优先。

**讨论**:
- PM: "Cursor 是目标用户的核心工具"
- QA: "外部依赖，测试困难"
- 开发者: "需要调研 Cursor 扩展 API，不确定工作量"
- 社区: "Cursor 社区是传播渠道"

**共识**: Cursor 插件保持 P1，但在 TUI 之后启动。先通过 MCP 集成验证需求，再决定是否开发独立插件。

## 九、执行原则

1. **体验优先** — 每个功能从用户视角验收，不是从技术视角
2. **债务随行** — 新功能必须偿还关联技术债务，不欠新债
3. **小步快跑** — 每个 Sprint 可独立交付，不跨 Sprint 依赖
4. **文档留痕** — 每个决策记录原因，每个变更记录影响
5. **保持轻量** — 不引入重依赖，不偏离个人工具定位

---

> **核心共识**: 先让产品"值得被使用"，再让产品"容易被安装"。  
> **一句话**: v0.8.0 的目标不是发布，而是让用户爱上 CarryMem。
